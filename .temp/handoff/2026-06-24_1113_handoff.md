## Goal

Train an RL policy (FastSAC) to do **peg insertion** on the Newton / mujoco_warp
peg env using an **operational-space controller (OSC)**, mirroring jax_rl's
FactoryPegInsert. The OSC was unstable for a long time (root cause now found +
fixed). Immediate aim: a retrain that actually *inserts*; bigger aim: **batch the
env** for jax_rl-class throughput, then a long run.

## What was tried and FAILED (so we don't redo it)

- **Wrong root-cause theories for the OSC instability — ALL were artifacts of the
  scrambled Jacobian:** "near-singular ARM_Q pose", weld compliance/stiffness,
  payload (peg mass) inertia, peg-bore contacts, the clip-box / action frame,
  gravity-comp sign, the DLS ridge, the nullspace term, fp32-vs-fp64. Don't chase
  these again — they move the symptom but aren't the cause.
- **`mjw.forward` staleness** was a *real* secondary bug (fixed) but did NOT fix the
  divergence — the Jacobian shape did.
- **Bounded-slew / EE-following action** was a workaround for the divergence; once
  the Jacobian was fixed, pure absolute (and delta) converge — slew not needed.
- **Batching: NOT attempted yet** (deliberately deferred — multi-file rewrite, not
  worth shipping unverified overnight).
- **Test-harness gotchas that bit me (avoid):** (1) dropping the `s0,s1` swap freezes
  the sim (looks like "error stuck"); (2) `pkill -f <name>` matches *my own command
  line* and self-kills the shell — never pkill a string that's in the running
  command; (3) `uv run` + `> file` redirects sometimes vanish / re-sync the env —
  use `.venv/bin/python` directly for probes & the recorder; (4) headless `ViewerGL`
  needs `DISPLAY=:1` (or EGL).
- GPU is **shared** (user3 ~6.7 GB). Keep `mem_fraction` modest; our train uses ~0.7 GB.
